# ADR 0010: Introduce Mockito for Unit Testing

## Status
- **Proposed** → **Accepted**

## Context
- The project currently uses **Spring Boot 3.2.5** with **JUnit Jupiter 5.10.2** (via `spring-boot-starter-test`).
- Mockito is already transitively pulled in (`mockito-core` **5.7.0**, `mockito-junit-jupiter` **5.7.0**) but is **not explicitly version‑pinned**.
- No JUnit‑4 tests are present (`@RunWith` annotations were not found).
- Existing test classpath includes **ByteBuddy 1.14.13** (used by Mockito) and **ASM** from other libraries.
- Future test code may require mocking final classes or static methods.

## Decision
1. **Explicitly declare Mockito dependencies** with fixed versions to avoid accidental upgrades through transitive changes.
2. Use **`mockito-junit-jupiter`** (JUnit‑5 extension) as the primary integration point because the codebase already uses JUnit Jupiter.
3. Add **`mockito-inline`** (optional, test‑scope) to enable mocking of final classes and static methods when needed.
4. Pin **ByteBuddy** to the same version as Mockito (1.14.13) via `<dependencyManagement>` to guarantee binary compatibility across the whole test classpath.
5. No JUnit‑4 migration is required; legacy `@RunWith(MockitoJUnitRunner.class)` will be avoided.

## Consequences
- All new and existing tests will import `org.junit.jupiter.api.Test` and use `@ExtendWith(MockitoExtension.class)`.
- The build will remain deterministic; Maven’s dependency resolution will not upgrade Mockito or ByteBuddy without a deliberate change.
- Test code can safely mock final classes by adding `mockito-inline` to the test dependencies.
